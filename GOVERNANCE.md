## Decision Making

### Day-to-Day Decisions
- Bug fixes and minor improvements can be submitted via PR without prior discussion
- PRs require at least one maintainer review before merging

### Major Decisions
For significant changes:
1. Open an issue to discuss the proposal
2. Allow time for community feedback (minimum 1 week)
3. Maintainers make the final decision

### Security Decisions
Security-related decisions are made by the Security Lead (@SHAdd0WTAka) with input from the security community.

## Contribution Acceptance Criteria

Contributions are evaluated based on:
- Code quality and adherence to style guidelines
- Test coverage for new features
- Documentation completeness
- Security considerations
- Backward compatibility

## Release Process

1. **Version Numbering**: We follow [Semantic Versioning](https://semver.org/)
2. **Release Cycle**: Releases are made as needed, not on a fixed schedule
3. **Release Notes**: All changes are documented in [CHANGELOG.md](CHANGELOG.md)
